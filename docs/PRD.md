# Food Sense V0.2 PRD

## 1. 产品概述

Food Sense 是一款面向食物敏感、食物不耐受，以及怀疑自己存在食物相关不适人群的手机网页产品。

它帮助用户回答三个日常问题：

1. 这个我能吃吗？
2. 我刚才/今天吃了什么？
3. 我现在不舒服，可能和之前吃过的什么有关？

产品通过个人食物档案、配料/菜单/食品检查、饮食记录、症状记录和简单的规律发现来完成这些任务。

## 2. 产品定位

Food Sense 是一个个人食物反应助手。

它帮助用户：

- 避开自己已知需要注意的食物或成分。
- 发现包装食品、餐厅菜单、外卖截图、酱料、加工食品里的隐藏成分。
- 轻松记录真实生活里的饮食，而不是做复杂的营养打卡。
- 在不舒服时快速记录症状。
- 回看不舒服前吃过什么。
- 随着使用时间增加，发现值得继续观察的食物规律。

Food Sense 不是医疗诊断产品，不替代医生建议，也不直接判断用户“对某个食物过敏”。

## 3. 目标用户

### 主要用户

已经知道自己对某些食物敏感或不耐受的人，通常来自医院或检测报告，但在日常饮食里仍然很难避开隐藏成分，也很难判断不舒服到底和哪顿饭有关。

例子：

- 用户报告里显示蔗糖、胡萝卜、黄瓜、乳制品或鸡蛋相关项目为弱阳性/阳性。
- 用户经常饭后腹痛，但无法快速判断是哪顿饭、哪个食材或哪个隐藏成分可能相关。

### 次要用户

没有医院报告，但经常饭后不舒服，希望通过记录饮食和症状来发现规律的人。

### 第一批试用用户

5-20 人，小范围私密试用。

## 4. 核心问题

1. 已知敏感食物很难避开，因为很多成分是隐藏的。
2. 用户不舒服后，经常不知道是哪顿饭或哪个食材可能相关。
3. 火锅、烧烤、聚餐、外卖这类复杂饮食很难记录。
4. 有些用户没有医院报告，产品仍然要能从记录开始发挥价值。
5. 产品需要有帮助，但不能做不安全的医疗判断。

## 5. V1 核心循环

```text
判断眼前食物能不能吃
-> 记录自己吃了什么
-> 不舒服时记录症状
-> 回看症状前的饮食
-> 长期发现可能相关的食物规律
```

## 6. V1 核心动作

### 6.1 能吃吗

用途：

- 拍包装食品配料表。
- 上传外卖截图。
- 拍餐厅菜单。
- 粘贴配料或菜单文字。
- 在超市查看零食、饮料、调料、酱料、加工食品。

结果应该包括：

- 是否命中用户个人食物档案里的项目。
- 命中了哪些原文，比如 `全脂乳粉`、`白砂糖`、`蛋黄酱`。
- 为什么提示这个项目。
- 如果信息不足，不假装给出确定结论，而是提示需要补充配料或询问商家。

### 6.2 记吃了什么

用途：

- 保存一顿饭，可以发生在吃前、吃饭时或吃完后。
- 保存零食、饮料、酱料、调料、单个食品。
- 用照片、菜单截图、外卖订单截图或一句话记录复杂饮食。

复杂餐处理：

- 不要求用户每上一道菜都记录一次。
- 可以拍一张桌面、菜单或订单截图。
- 可以补一句话，例如 `火锅：牛肉、虾滑、豆腐、青菜、蘸料`。
- 产品可以轻量提示用户补充 `锅底`、`蘸料`、`饮料`、`甜品`、`加工食品`。

### 6.3 记录不舒服

用途：

- 记录腹痛、腹胀、腹泻、恶心、皮肤反应等不适。
- 记录开始时间、严重程度、持续时间和备注。
- 保存后立即回看过去 2、4、8、24 小时内的饮食记录。

## 7. V1 功能范围

### 7.1 个人食物档案

用户可以维护对自己重要的食物或成分。

每个项目可以包含：

- 食物/成分名称
- 别名或相关成分
- 状态：需要避开、正在观察、不确定
- 严重程度或报告等级
- 检测值，如有
- 来源：医院报告、手动添加、产品发现、用户观察
- 备注

这个区域的产品命名还没有最终确定。`清单`、`关注`、`避雷`、`忌口`、`我的食物` 都还不够理想。

### 7.2 报告照片提取

有医院或检测报告的用户可以上传报告照片。

产品需要提取：

- 食物名称
- 检测数值
- 结果等级，例如阴性、弱阳性、阳性
- 来源信息

提取结果必须由用户确认后，才进入个人食物档案。

### 7.3 手动添加食物

没有报告的用户也可以手动添加食物或成分。

例子：

- 牛奶
- 鸡蛋
- 蔗糖
- 胡萝卜
- 辛辣
- 油炸食物

手动添加的项目可以标记为：需要避开、正在观察、不确定。

### 7.4 食物检查

用户可以检查食品、包装、菜单、截图或文字。

V1 输入方式：

- 拍照
- 上传图片
- 粘贴文字

V1 输出：

- 命中的个人食物项目
- 命中的配料或菜单原文
- 提示原因
- 下一步建议，例如避开、谨慎、询问商家、保存为记录

### 7.5 饮食记录

用户可以记录吃过或准备吃的东西。

字段：

- 时间
- 照片或截图
- 文字描述
- 提取出的食物标签
- 场景：家里、餐厅、外卖、超市、其他
- 可选标记：酱料、调料、饮料、甜品、加工食品

### 7.6 症状记录

用户可以记录不舒服。

字段：

- 开始时间
- 症状类型
- 严重程度
- 持续时间
- 备注

保存症状后，产品展示相关时间窗口内的饮食记录。

### 7.7 发现

产品展示非诊断性质的规律：

- 哪些食物经常出现在不舒服之前。
- 哪些个人食物项目最近经常被命中。
- 哪些餐次值得回看。
- 哪些食物建议继续观察。

措辞要谨慎：

- 可以说：`可能相关`、`建议继续观察`、`可在复诊时与医生讨论`。
- 不说：`你对这个过敏`、`这个一定导致了不舒服`。

## 8. V1 暂不做

V1 不做：

- 医疗诊断
- 判断某个食物一定导致症状
- 完整饮食计划
- 营养摄入分析
- 医生后台
- 社区分享
- 条形码数据库
- 大型公共食物数据库
- 多设备同步
- 公开发布
- 付费功能

## 10. 隐私原则

V1 先采用本地存储。

原则：

- 用户数据尽量存在用户自己的设备上。
- 不收集真实姓名、手机号、身份证号等非必要信息。
- 饮食、症状和检测报告数据都按敏感数据对待。
- 如果 AI 识别需要把图片或文字发送给外部服务，必须在上传前明确告知用户。
- 优先保存结构化结果，不长期保存原始医院报告图片。
- 提供一键删除所有本地数据的能力。
- 产品内要有一页简短易懂的隐私说明。

## 11. 私测成功标准

V1 小范围试用成功的标准：

- 没有报告的用户也能开始使用。
- 有报告的用户可以提取并确认关键食物项目。
- 用户可以检查食品/菜单/包装，并理解为什么被提示。
- 简单饮食记录可以在 30 秒内完成。
- 不舒服记录可以在 30 秒内完成。
- 记录不舒服后，用户能看到相关时间窗口内的饮食。
- 使用几天后，产品能展示基础的重复出现规律。

## 12. 未决问题

1. 个人食物档案区域到底叫什么？
2. `能吃吗` 的结果应该是简单答案，还是分成适合/谨慎/信息不足？
3. 如何表达不确定性，既不误导用户，也不让用户觉得没有价值？
4. 报告原图是否保存在本地，还是只保存结构化结果？
5. 复杂餐自动提取后，用户如何轻松编辑？
6. 发现规律需要多少数据才值得提示？
7. 产品视觉气质应该更偏临床、平静生活化，还是温暖助手型？

## 13. MVP 定义

MVP 的目标不是一次做完整健康管理产品，而是验证一个最小但真实可用的闭环：

```text
用户能建立自己的食物信息
-> 能检查眼前食物
-> 能记录吃过什么
-> 能记录不舒服
-> 能回看并发现基础规律
```

V1 第一轮开发切片见 `docs/v1-mvp-slice.md`。真实用户路径走查见 `docs/v1-flow-walkthrough.md`。第一轮建议先做本地可用闭环，把 AI 图片识别和报告 OCR 放到核心手动流程跑通之后。

### 13.1 MVP 必须做

1. 本地存储用户数据。
2. 支持手动添加个人食物项目。
3. 支持从报告图片中提取食物项目，并让用户确认。
4. 支持 `能吃吗`：拍照/上传图片/粘贴文字，识别并匹配个人食物项目。
5. 支持 `记吃了什么`：用图片、文字和标签记录食物。
6. 支持 `记录不舒服`：记录时间、类型、严重程度、持续时间和备注。
7. 记录症状后，展示过去 2、4、8、24 小时内的饮食。
8. 展示基础 `发现`：哪些食物或标签经常出现在不舒服前。
9. 提供本地数据删除入口。
10. 在涉及 AI 上传图片/文字前，提示用户数据会被发送用于识别。

### 13.2 MVP 可以简化

1. 不做账号登录。
2. 不做云端同步。
3. 不做复杂营养分析。
4. 不做正式医生报告导出。
5. 不做大型食物数据库。
6. 不做支付和公开发布。
7. 不做多语言，先只做中文。

### 13.3 MVP 验收标准

1. 新用户 1 分钟内能完成第一条饮食记录。
2. 用户能在 30 秒内记录一次不舒服。
3. 有报告的用户能从报告截图中提取至少一批候选食物项目。
4. 用户能看到提取结果，并手动确认/删除/修改。
5. 用户上传配料表后，能看到命中的个人食物项目和命中原因。
6. 用户记录不舒服后，能看到之前相关时间窗口内的饮食。
7. 连续使用几天后，用户能看到基础的重复出现提示。

## 15. 核心流程

### 15.1 有报告用户首次使用

```text
进入产品
-> 打开个人食物档案
-> 上传报告图片
-> AI 提取食物名称、数值、等级
-> 用户确认/编辑
-> 保存为个人食物项目
-> 后续用于 `能吃吗` 和记录分析
```

### 15.2 无报告用户首次使用

```text
进入产品
-> 不强制建档
-> 直接开始 `记吃了什么`
-> 不舒服时 `记录不舒服`
-> 使用几天后在 `发现` 查看可能相关食物
-> 用户可把某些食物加入个人食物档案
```

### 15.3 能吃吗流程

```text
点 `+`
-> 选择 `能吃吗`
-> 拍照/上传/粘贴文字
-> 识别文字或图片内容
-> 匹配个人食物档案和别名
-> 展示命中结果
-> 用户选择保存为检查记录或饮食记录
```

### 15.4 记吃了什么流程

```text
点 `+`
-> 选择 `记吃了什么`
-> 添加图片/截图/文字
-> 选择或确认时间
-> 补充场景和标签
-> 产品提取食物标签
-> 用户确认
-> 保存饮食记录
```

### 15.5 记录不舒服流程

```text
点 `+`
-> 选择 `记录不舒服`
-> 选择症状类型
-> 选择严重程度
-> 确认开始时间和持续时间
-> 保存
-> 产品展示过去 2、4、8、24 小时饮食
-> 用户标记值得观察的食物
```

### 15.6 发现流程

```text
用户积累饮食和症状记录
-> 产品统计症状前反复出现的食物/标签
-> 排除数据不足的项目或降低提示强度
-> 展示 `可能相关` 和 `建议继续观察`
-> 用户可查看关联记录
```

## 16. 技术方案

这是 V1 技术方案草案，后续开发前还可以调整。

### 16.1 产品形态

V1 建议做手机网页 / PWA。

原因：

- 适合 5-20 人快速私测。
- 不需要 App Store 或小程序审核。
- 可以通过链接分享。
- 可以支持拍照、上传图片和本地存储。

### 16.2 前端

建议：

- React + TypeScript
- Vite
- 移动优先的响应式界面
- PWA 能力后续再加，不作为第一天必须项

原因：

- 开发速度快。
- 适合构建交互型手机网页。
- 后续可逐步扩展。

### 16.3 数据存储

V1 使用浏览器本地存储。

建议：

- 结构化数据使用 IndexedDB。
- 简单设置可以使用 localStorage。
- 图片是否长期保存需要单独确认，默认优先保存结构化结果。

本地数据类型：

- 个人食物项目
- 检查记录
- 饮食记录
- 症状记录
- 发现结果缓存
- 用户设置

### 16.4 AI 识别

V1 可能需要 AI 处理：

- 医院报告图片提取
- 配料表 OCR 和结构化
- 菜单/外卖截图理解
- 饮食记录标签提取

建议方式：

- 前端调用一个轻量后端接口，由后端转发到 AI 服务。
- 不建议把 API Key 放在前端。

如果第一阶段只做本地演示，可以先用 mock 结果或手动输入流程验证产品体验，再接 AI。

### 16.5 后端

V1 私测如果要接真实 AI，需要一个很薄的后端。

后端职责：

- 接收图片或文字
- 调用 AI 识别服务
- 返回结构化结果
- 不保存用户数据，或默认不持久化

可以选：

- Next.js API Routes
- Vercel Serverless Functions
- 简单 Node/Express 服务

### 16.6 部署

私测阶段建议：

- 前端部署到 Vercel 或 Netlify。
- 如果需要 AI 后端，可以同部署在 Vercel。
- 数据仍保存在用户本地浏览器。

### 16.7 未来升级方向

后续如果要公开发布，可能需要：

- 账号系统
- 云端同步
- 数据导出
- 更完整的隐私政策和用户协议
- 更严格的数据安全设计
- 更稳定的 AI 识别和人工确认流程

## 17. 主要风险与应对

### 17.1 医疗误导风险

风险：

- 用户可能把产品提示理解成医疗诊断。

应对：

- 明确声明产品不做诊断。
- 用 `可能相关`、`建议观察`、`建议咨询医生`。
- 避免 `你过敏`、`一定导致` 等措辞。

### 17.2 AI 识别错误风险

风险：

- OCR 或 AI 可能漏识别、误识别配料、菜单或报告内容。

应对：

- 所有报告提取结果必须用户确认。
- 展示命中原文，让用户能检查依据。
- 对信息不足的场景提示补充信息。
- 不把 AI 结果包装成绝对结论。

### 17.3 隐私风险

风险：

- 报告、症状和饮食记录属于敏感数据。

应对：

- V1 本地存储。
- 不要求账号。
- 上传 AI 前明确提示。
- 提供删除所有数据。
- 默认不长期保存报告原图。

### 17.4 用户记录负担过高

风险：

- 如果每顿饭都要详细填很多字段，用户会放弃。

应对：

- 允许一张图加一句话。
- 复杂餐按一条记录处理。
- 可选标签轻量补充，不做强制问卷。

### 17.5 规律发现不可靠

风险：

- 数据太少时，产品可能提示没有意义的相关性。

应对：

- 设置最低记录数量或降低提示强度。
- 用 `值得观察` 而不是 `原因`。
- 允许用户查看原始关联记录。

### 17.6 本地存储丢失风险

风险：

- 用户清理浏览器缓存或换手机后，数据可能丢失。

应对：

- 在隐私与数据页明确说明。
- 后续加入导出/导入。
- 云端同步放在后续版本评估。
